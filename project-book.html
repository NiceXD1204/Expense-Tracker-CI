מבוא

בעשור האחרון חלה קפיצה משמעותית באופן שבו מערכות תוכנה מפותחות, נפרסות ומתוחזקות. המעבר מעבודה מקומית על שרתים בודדים לתשתיות ענן מבוזרות, יחד עם עלייה בדרישות המשתמשים למהירות תגובה, זמינות גבוהה ויכולת שדרוג מיידית, הוביל לצורך במתודולוגיות עבודה חדשות.

אחת המתודולוגיות הבולטות בהקשר זה היא DevOps, המשלבת בין תהליכי פיתוח (Development) לבין תהליכי תפעול ותחזוקה (Operations), תוך דגש על אוטומציה, ניהול גרסאות, ניטור רציף ושיפור מתמיד. DevOps משנה את צורת החשיבה הארגונית בכך שהיא מקדמת עבודה שיתופית ואינטגרטיבית, במקום מודל היררכי ומבודד בו צוותים פועלים בנפרד.

פרויקט Expense Tracker משקף יישום מלא של עקרונות אלו. מדובר במערכת אינטרנטית לניהול הוצאות והכנסות אישיות, אשר פותחה תוך הקפדה על מודולריות, אבטחת מידע, ניהול עלויות מושכל ותפעול יציב. הפרויקט כולל הקמה מלאה של תשתית ענן ב-AWS, קונטיינריזציה של כל רכיבי המערכת, אשכול Kubernetes מנוהל (EKS), פריסה בשיטת GitOps באמצעות ArgoCD, ותהליכי CI/CD אוטומטיים מלאים באמצעות GitHub Actions.

ייחוד הפרויקט הוא בגישה שאומצה לניהול עלויות: מאחר שהתשתית מיועדת לסביבת פיתוח ולימוד ולא לשימוש מסחרי מתמשך, התשתית תוכננה מלכתחילה כ"ephemeral" – ניתנת להקמה ולהריסה מלאה תוך דקות באמצעות Terraform, כך שהעלות נושאת רק בזמן שהסביבה אכן פעילה.

באמצעות פרויקט זה, מתקבל מודל עבודה המתאים לעולמות התעשייה כיום, ומהווה דוגמה לאופן שבו ניתן להעביר אפליקציה קיימת לסביבת ענן מאובטחת, מנוהלת ומבוקרת – תוך שמירה על משמעת עלויות מחמירה, המתאימה לפרויקט לימודי.

רקע תאורטי

ארכיטקטורת Monolith לעומת Microservices

במערכות מונוליטיות, כל רכיבי האפליקציה מאוגדים יחד בקוד אחד: ממשק המשתמש, הלוגיקה העסקית והתקשורת למסד הנתונים. מצב זה מקל על פיתוח ראשוני, אך יוצר קושי עצום ככל שמורכבות המערכת גדלה:

  • כל שינוי קטן דורש הידור מחדש של המערכת כולה
  • איתור תקלות מורכב וארוך
  • קיים קושי לבצע Scaling נקודתי
  • תלות בין רכיבים מקשה על הוספת פיצ'רים באופן עצמאי

כתגובה, התפתחה ארכיטקטורת Microservices, המחלקת את האפליקציה למספר שירותים עצמאיים. כל שירות מנהל אחריות תחומה, ניתן לפיתוח, בדיקה ופריסה עצמאית, ומסוגל לגדול בהתאם לצריכה שלו.

מבנה זה משפר:

  • זמני פיתוח
  • זמינות מערכתית
  • אפשרות ליישום טכנולוגיות שונות באותה מערכת
  • עמידות בפני תקלות

בפרויקט Expense Tracker, האפליקציה חולקה לשלושה שירותים עצמאיים: שירות Frontend (React), שירות Backend (FastAPI) ומסד נתונים (PostgreSQL) – כל אחד ארוז כ-container נפרד, נפרס כ-Deployment עצמאי ב-Kubernetes, וניתן לעדכון ולהרחבה ללא השפעה על השירותים האחרים.

DevOps – חיבור בין פיתוח לתפעול

בעבר, צוותי פיתוח תפקידם היה כתיבת קוד בלבד, בעוד צוותי תפעול התמקדו בפריסה, תחזוקה וניהול שרתים. מודל זה יצר פער משמעותי בין הסביבה בה מפתחים לבין זו שבה המערכת רצה בפועל.

DevOps שואפת לצמצום פער זה על ידי:

  • אוטומציה של תהליכי Build, Test ו-Deployment
  • שימוש בכלים לניהול קוד, ניטור ובקרת תצורה
  • יצירת Pipeline אחיד משלב כתיבת הקוד ועד ההפעלה בסביבת הענן
  • שיתוף פעולה ותקשורת זמינה בין הצוותים

התוצאה: שחרור גרסאות מהיר יותר, אמין יותר ומבוקר יותר.

Dev ──► Plan ──► Code ──► Build ──► Test ──► Release ▲ │ │ ▼ Monitor ◄── Operate ◄── Deploy ◄──────────────── Ops

CI/CD – אינטגרציה ופריסה רציפה

תהליך Continuous Integration (CI) מבטיח שכל שינוי קוד עובר בדיקות אוטומטיות טרם שילובו בענף הראשי.

תהליך Continuous Deployment/Delivery (CD) דואג לפריסה אוטומטית של הקוד לסביבה התפעולית, ללא צורך בהתערבות ידנית.

בפרויקט Expense Tracker:

  • CI בוצע באמצעות GitHub Actions – lint (ruff), בדיקות (pytest) ובנייה (Docker build)
  • CD בוצע באמצעות ArgoCD בשיטת GitOps
  • פריסת הסביבה נעשתה באמצעות Kubernetes + Helm על גבי EKS
  • התשתית נבנתה כולה באמצעות (IaC) Terraform
  • אימות מול AWS בוצע ללא מפתחות סטטיים – באמצעות GitHub OIDC ותפקיד IAM ייעודי
בחירה מרכזית בפרויקט הייתה שימוש ב-OIDC (OpenID Connect) לצורך אימות GitHub Actions מול AWS, במקום שמירת מפתחות גישה סטטיים כ-Secrets. גישה זו מפחיתה משמעותית את שטח התקיפה, שכן אין כלל אישורים קבועים שניתן לגנוב או לדלוף.

ארכיטקטורת המערכת הכללית (Microservices)

האתר מורכב באמצעות Docker, בארכיטקטורת Microservices.

  • המערכת מבוססת על חלוקה לשירותים עצמאיים, שכל אחד אחראי על תחום לוגי מסוים, כגון הצגת הממשק, ניהול לוגיקה עסקית או אחסון נתונים. גישה זו מאפשרת פיתוח ותחזוקה יעילים לאורך זמן.
  • כל שירות ניתן לפריסה, הרחבה או החלפה באופן עצמאי, ללא תלות בשירותים אחרים. תכונה זו מייצרת גמישות גבוהה ומאפשרת תגובה לצרכים עסקיים משתנים.

שכבת ה-Frontend

שכבת ה-Frontend נבנתה באמצעות React ו-Vite ככלי הבנייה (build tool), ומוגשת בסביבת הייצור על גבי קונטיינר nginx.

  • NGINX הוא שרת אינטרנט (web server) חזק ומהיר המשמש גם כ-reverse proxy ו-load balancer. הקונטיינר אחראי על הגשת קבצים סטטיים, יצירת חוויית המשתמש, והפניית בקשות API אל שכבת ה-Backend.
  • התקשורת מול השרת מבוססת על קריאות API מאובטחות תחת נתיב /api.

הקובץ Dockerfile של הפרונטנד בנוי בשיטת Multi-stage build:

  1. שלב ראשון (builder) – מבוסס node, מתקין תלויות ומריץ npm run build
  2. שלב שני (production) – מבוסס nginx:alpine, מעתיק את תוצרי הבנייה בלבד

גישה זו מייצרת image סופי קטן במיוחד, שכן הוא אינו כולל את כלי הבנייה, node_modules או קוד המקור – רק את הקבצים הסטטיים הדרושים להרצה.

שכבת ה-Backend

הבקאנד בנוי על גבי FastAPI (Python) – פריימוורק מודרני, מהיר וקל־משקל לבניית ממשקי API, המנהל את כל הלוגיקה העסקית של ניהול ההוצאות וההכנסות, כולל אימות משתמשים מבוסס JWT וכתיבה/קריאה ממסד הנתונים.

קובץ ה-Docker Compose המשמש לפיתוח מקומי ממחיש את שלושת השירותים ואת האופן שבו הם מתקשרים:

services:
  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: expenses
      POSTGRES_PASSWORD: expenses   # local dev only - never use this in the cloud
      POSTGRES_DB: expenses
    ports:
      - "5432:5432"
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U expenses"]
      interval: 5s
      timeout: 3s
      retries: 5

  backend:
    build: ./backend
    environment:
      DATABASE_URL: postgresql://expenses:expenses@db:5432/expenses
      JWT_SECRET_KEY: local-dev-secret-change-before-going-to-production
    ports:
      - "8000:8000"
    depends_on:
      db:
        condition: service_healthy

  frontend:
    build:
      context: ./frontend
      args:
        VITE_BACKEND_URL: /api
    environment:
      BACKEND_URL: http://backend:8000
    ports:
      - "8501:80"
    depends_on:
      - backend

volumes:
  pgdata:

לצורך בדיקות איכות קוד, נעשה שימוש ב-ruff (linting מהיר לפייתון) וב-pytest לבדיקות יחידה. בנוסף, שולב prometheus-fastapi-instrumentator המאפשר חשיפת מדדי ביצועים ישירות מהאפליקציה עבור Prometheus.

קונטיינריזציה באמצעות Docker ו-ECR

Docker – אריזת שירותים ליחידות הרצה עקביות

  • Docker מאפשר לארוז כל שירות ליחידת הרצה עצמאית הכוללת את התלויות הדרושות לו. כך ניתן להבטיח שהשירות יתנהג באופן זהה בכל סביבה — פיתוח, בדיקות וייצור.
  • הפרדה זו מאפשרת שחזור מהיר, פריסה אחידה ופתרון קל לתקלות שנובעות מהבדלים בסביבה.

ECR – מאגר Images מנוהל

  • כל ה-Images שנבנים בתהליך ה-CI מאוחסנים ב-ECR (Amazon Elastic Container Registry) של AWS, המבטיח אבטחת גישה, שליטה בגרסאות ויעילות בשרשרת הפריסה.
  • המאגר מאפשר ניהול מרכזי מסודר והפחתת סיכונים כתוצאה מהחזקת גרסאות מפוזרות.
  • המאגר מספק נקודת משיכת תמונות דוקר ל-EKS – צמתי ה-worker מושכים את התמונה ישירות באמצעות תפקיד IAM המחובר לצומת, ללא צורך באישורים נוספים.
Dockerfile ──► docker build ──► Docker Image ──► docker push ──► ECR ──► EKS pulls image

תזמור השירותים באמצעות Kubernetes

Kubernetes – מערכת התזמור

  • Kubernetes מנהלת את הקונטיינרים על ידי הרצה מאוזנת, החלמה אוטומטית (Self-Healing), עדכונים הדרגתיים (Rolling Updates) ויכולת ביצוע Scale דינמי בהתאם לעומס בפועל.
  • המערכת נשארת זמינה, יציבה ובעלת יכולת התאוששות גם מול תקלות בלתי צפויות.

הקלאסטר עצמו הוא Amazon EKS (Elastic Kubernetes Service) – שירות Kubernetes מנוהל של AWS, הפרוס על גבי VPC ייעודי בשתי רשתות משנה פרטיות.

Helm – package manager for k8s

Helm הוא כלי לניהול חבילות עבור Kubernetes, שנועד לפשט את תהליך הפריסה והניהול של יישומים מורכבים. הכלי עושה שימוש במבנים הנקראים Charts, המכילים את כל קבצי ההגדרות הדרושים להפעלת היישום, ובכך מספק דרך מאורגנת, אחידה וגמישה לניהול סביבת Kubernetes.

בפרויקט נוצרו שלושה Charts, אחד לכל שירות: backend, frontend ו-db. כל אחד מגדיר Deployment, Service, Ingress (במידת הצורך) ואת דרישות המשאבים שלו.

דוגמה אמיתית מקובץ charts/backend/values.yaml:

resources:
  requests:
    cpu: 100m
    memory: 128Mi
  limits:
    cpu: 250m
    memory: 256Mi

# Extra environment variables for the backend container (e.g. DATABASE_URL).
# Do NOT put secrets here - this ends up in the Helm release values/history in
# plain text. Use envSecretName below for anything sensitive.
env: {}

הגדרת requests ו-limits לכל שירות מבטיחה שלא ייווצר מצב בו pod יחיד "בולע" את כל משאבי הצומת (node) ומקריס שירותים אחרים - עיקרון מרכזי בסביבת ריצה משותפת.

יתרונות Helm בפרויקט

  • אחידות בפריסות בין סביבות (dev, staging, production)
  • קובץ values.yaml נפרד לכל סביבה, ללא שינוי במבנה הכללי של הקוד
  • יכולת rollback מהירה במקרה של תקלה
  • שילוב עם ArgoCD, כך שכל שינוי ב-values מסתנכרן אוטומטית לאשכול

ניהול תשתיות (Infrastructure as Code) באמצעות Terraform

אחד מהעקרונות המרכזיים של הפרויקט היה ניהול תשתיות הענן בצורה עקבית, ניתנת לשחזור ובעלת רמת בקרה גבוהה. Terraform מאפשר להגדיר את התשתיות בצורה הצהרתית: במקום להגדיר פעולות, מגדירים את התוצאה הרצויה, והכלי דואג לבצע את השלבים הדרושים.

מבנה דו-שכבתי: bootstrap מול envs/dev

התשתית חולקה לשתי שכבות נפרדות בריפוזיטורי Expense-Tracker-CD:

  • bootstrap – שכבה קבועה, המכילה את כל מה שצריך לשרוד הריסה (destroy) של הסביבה: דלי S3 לאחסון ה-state, טבלת DynamoDB לנעילת state, ו-Hosted Zone ב-Route53.
  • envs/dev – שכבה זמנית (ephemeral), הכוללת את האשכול עצמו, כל תוספי ה-Helm, וסודות האפליקציה. שכבה זו נהרסת ונבנית מחדש בכל מחזור עבודה.
terraform output (bootstrap) ──────────────────────────── bucket = "expense-tracker-tfstate-<account-id>" key = "envs/dev/terraform.tfstate" region = "eu-central-1" dynamodb_table = "expense-tracker-tf-locks" encrypt = true

שימוש במודולים קיימים

בהתאם לעיקרון של הימנעות מכתיבת קוד מיותר, נעשה שימוש במודולי Terraform רשמיים ומתוחזקים מהקהילה, ולא בכתיבה מאפס:

  • terraform-aws-modules/vpc/aws – בניית ה-VPC, רשתות משנה ציבוריות ופרטיות, NAT Gateway
  • terraform-aws-modules/eks/aws – הקמת אשכול EKS מלא כולל צמתי worker
  • terraform-aws-modules/iam//modules/iam-role-for-service-accounts-eks – יצירת תפקידי IAM עבור Service Accounts (IRSA), למשל עבור ExternalDNS

יתרונות הבחירה ב-Terraform

  1. שחזור סביבה מלא — כל התשתיות ניתנות להקמה מלאה מחדש מתוך הקוד בלבד
  2. שליטה בגרסאות — כל שינוי בתשתית מתועד בקבצי Git, כולל היסטוריה מלאה
  3. מניעת טעויות ידניות — אין צורך להקים משאבים באמצעות ממשק הענן
  4. יכולת הרחבה — הוספת שירותים חדשים לסביבה נעשית בצורה מסודרת ושקופה

רשת (Networking) – VPC, Ingress, DNS ו-TLS

VPC ורשתות משנה

האשכול פרוס בתוך VPC ייעודי, עם רשתות משנה ציבוריות (public subnets) לרכיבים החשופים לאינטרנט (כגון ה-Load Balancer), ורשתות משנה פרטיות (private subnets) לצמתי ה-EKS עצמם, כך שהם אינם נגישים ישירות מהאינטרנט.

Internet ──► Internet Gateway │ ┌──────┴───────┐ Public Subnet Public Subnet (NAT Gateway) (NAT Gateway) │ │ Private Subnet Private Subnet (EKS Node) (EKS Node)

Ingress Controller

  • ingress-nginx מותקן כתוסף Helm ומנהל את נתיבי התעבורה למיקרו-שירותים בתוך הקלאסטר.
  • מאפשר הפניית בקשות ל-Frontend ול-Backend על פי חוקים (host/path), ללא שינוי בקוד השירות.
  • משפר אבטחה בכך שאינו חושף שירותים פנימיים ישירות החוצה.

DNS אוטומטי – ExternalDNS

ExternalDNS עוקב אחר משאבי Ingress באשכול, וכותב רשומות DNS מתאימות ישירות אל ה-Hosted Zone ב-Route53, ללא צורך בעדכון ידני. תפקיד ה-IAM שלו מוגבל (IRSA) ל-Zone הספציפי בלבד – עיקרון least privilege.

TLS אוטומטי – cert-manager

cert-manager מנפיק אוטומטית תעודות TLS מ-Let's Encrypt באמצעות אתגר HTTP-01 המתבצע דרך ה-Ingress Controller עצמו, כך שכל כתובת מקבלת תעודה תקפה ללא כל פעולה ידנית וללא צורך בהרשאות DNS נוספות.

GitOps ו-ArgoCD — פריסה מבוססת אמת אחת

בפרויקט הוטמעה מתודולוגיית GitOps, בה ה-Git משמש כמקור האמת היחיד לכל מצב המערכת. כל שינוי בפריסה, בקונפיגורציה או בגרסת האפליקציה חייב להיות משוקף ב-Git, ומשם הוא מתגלגל אל סביבת הענן.

תבנית "App of Apps"

בפרויקט מוגדרת Application אחת מרכזית ("root") הצופה בתיקיית gitops/dev בריפוזיטורי ה-CD. Application זו, בתורה, אחראית ליצור ולנהל שלוש Applications בנות: אחת לכל שירות (backend, frontend, db). כל Application-בת שואבת את ה-Helm chart שלה מריפוזיטורי ה-CI, ואת קובץ ה-values שלה מריפוזיטורי ה-CD.

Expense-Tracker-CD/gitops/dev/ ├── backend-app.yaml ├── frontend-app.yaml ├── db-app.yaml ├── values-backend.yaml ◄── מתעדכן אוטומטית ע"י GitHub Actions └── values-frontend.yaml ◄── מתעדכן אוטומטית ע"י GitHub Actions

מדיניות הסנכרון מוגדרת כאוטומטית:

syncPolicy:
  automated:
    prune: true
    selfHeal: true
  syncOptions:
    - CreateNamespace=true
  • prune – מסירה מהאשכול משאבים שהוסרו מ-Git
  • selfHeal – מחזירה שינוי ידני שנעשה ישירות באשכול בחזרה למצב המוגדר ב-Git

כך מובטחת עקביות מלאה בין הקוד לבין מצב האשכול בפועל, שקיפות תפעולית מלאה, והפחתה דרמטית בטעויות אדם.

CI/CD – מתודולוגיית GitHub Actions

זרימת העבודה בפרויקט מבוססת על GitHub Flow: מפתח פותח ענף feature, לאחר שינוי ובדיקות מוצלחות נפתח Pull Request, ולאחר מיזוג לענף ה-master מתבצעת פריסה אוטומטית לסביבת ה-staging. תיוג גרסה (tag בפורמט v*.*.*) מפעיל פריסה לסביבת הייצור.

שרשרת ה-Workflows

קובץטריגרפעולה
ci.ymlPull Request לענף masterlint (ruff) + test (pytest) + build (docker)
deploy-staging.ymlpush לענף masterבונה ודוחפת images ל-ECR, מעדכנת values ב-CD repo, ArgoCD מסנכרן
deploy-production.ymlpush של tag בפורמט v*.*.*אותו תהליך, עם תיוג הגרסה הספציפית

קטע מתוך workflow הבדיקה המשותפת (_reusable-ci.yml):

jobs:
  lint:
    steps:
      - run: ruff check backend frontend

  test:
    steps:
      - run: pip install -r backend/requirements.txt
      - working-directory: backend
        run: pytest -v

  build:
    strategy:
      matrix:
        service: [backend, frontend]
    steps:
      - uses: docker/build-push-action@v6
        with:
          context: ${{ matrix.service }}
          push: false

אימות מול AWS ללא מפתחות סטטיים

ה-workflow משתמש ב-aws-actions/configure-aws-credentials עם role-to-assume, המבוסס על OIDC. כלומר, GitHub Actions "מתחזה" זמנית לתפקיד IAM ייעודי לפעולת ה-deploy, ללא צורך לאחסן Access Key ו-Secret Key קבועים כ-Secrets בריפוזיטורי.

עדכון אוטומטי של ריפוזיטורי ה-GitOps

לאחר בניית התמונה ודחיפתה ל-ECR, ה-workflow מבצע checkout לריפוזיטורי ה-CD, מעדכן את תגית התמונה בקובץ ה-values המתאים, ומבצע commit ו-push חזרה. ArgoCD מזהה את השינוי תוך דקות ומבצע את הפריסה בפועל.

התראות

בסיום כל ריצה (הצלחה או כישלון) נשלחת הודעה אוטומטית לערוץ Slack ייעודי, הכוללת קישור ישיר להרצה ב-GitHub Actions.

ניטור – Prometheus, Grafana ו-Loki

Prometheus

Prometheus הוא כלי קוד פתוח לניטור ואיסוף מדדים בזמן אמת, המבוסס על מנגנון Pull-Based, שבו השרת שולף נתונים מ-Exporters ומאחסן אותם כמערכי Time-Series. בפרויקט, Prometheus הותקן באמצעות ה-chart kube-prometheus-stack כתוסף Terraform, ואוסף מדדים על עומסים, זמינות שירותים וצריכת משאבים בקלאסטר.

Grafana

Grafana מספקת שכבת תצוגה ויזואלית לנתוני Prometheus, ומאפשרת בניית לוחות בקרה הכוללים גרפים, טבלאות והתראות (Alerts).

Alertmanager והתראות Slack

Alertmanager מוגדר לנתב כל התראה ברמת warning ומעלה לערוץ Slack ייעודי, כולל Pod crash-loop וכל תקלה אחרת שקיימת עבורה כלל ברירת מחדל ב-kube-prometheus-stack.

Loki + Grafana Alloy – ריכוז לוגים

בנוסף למדדים (metrics), נדרש פתרון לריכוז לוגים (logs) ממערכת הקונטיינרים. בפרויקט נבחר Grafana Loki במצב Monolithic (single-binary), יחד עם Grafana Alloy כסוכן איסוף הלוגים מכל pod באשכול (מחליף את Promtail, שהופסקה תחזוקתו). Loki מוגדר כמקור נתונים (datasource) נוסף באותה Grafana הקיימת, כך שמדדים ולוגים נצפים באותו ממשק בדיוק: 

grafana:
  additionalDataSources:
    - name: Loki
      type: loki
      access: proxy
      url: http://loki.monitoring.svc.cluster.local:3100

שילוב זה נבחר במקום מחסנית Elasticsearch/Kibana המסורתית, מאחר שהוא קל-משקל יותר (single pod, ללא PVC), ואינו דורש רכיבי ניהול נוספים - התאמה טובה לסביבת ריצה זמנית (ephemeral) שנהרסת ונבנית מדי יום.

אבטחת מידע

ניהול סודות

  • מפתח ה-JWT נוצר אוטומטית באמצעות random_password ב-Terraform, ונשמר כ-Kubernetes Secret - הוא אינו מופיע בשום שלב בקוד או בקובץ values גלוי.
  • קבצי terraform.tfvars ו-backend.hcl, המכילים פרטי סביבה רגישים, מוגדרים ב-.gitignore ואינם מגיעים לריפוזיטורי כלל.
  • Webhook כתובת ה-Slack מוזרמת כ-Secret נפרד ל-Alertmanager, ולא כערך גלוי בקובץ values.

הרשאות מצומצמות (Least Privilege)

לצורך הרצת Terraform (בין אם ע"י מפתח אנושי ובין אם דרך GitHub Actions), הוגדרה מדיניות IAM מותאמת אישית וממוקדת - ולא הרשאת AdministratorAccess גורפת. המדיניות כוללת הרשאות נפרדות וממוקדות עבור: EC2/VPC, EKS, ECR, ELB, IAM (מוגבל לפעולות הדרושות בלבד), S3 (מוגבל לדליים בעלי קידומת expense-tracker-*), DynamoDB (מוגבל לטבלת ה-locks), KMS, CloudWatch/Logs, ו-Route53 (הרשאות קריאה בלבד, לצורך שאילתת ה-Hosted Zone הקיים).

הצפנה

  • כל התעבורה החיצונית מוצפנת ב-HTTPS/TLS, באמצעות תעודות Let's Encrypt המתחדשות אוטומטית.
  • דלי ה-S3 המשמש לאחסון state מוגדר עם הצפנה (Server-Side Encryption) ומניעת גישה ציבורית.

בדיקות אוטומטיות כשכבת הגנה

כל שינוי קוד עובר lint ובדיקות יחידה לפני שהוא יכול להתמזג לענף הראשי, מה שמונע הגעת קוד פגום או לא בטוח לסביבת הענן.

ניהול עלויות תשתית ענן

שיקול מרכזי בתכנון הפרויקט היה עלות ההרצה, מאחר שמדובר בסביבת לימוד ולא בסביבת ייצור מסחרית. אשכול EKS, למשל, גובה עלות שעתית עבור ה-control plane בלבד, ללא תלות בשימוש בפועל.

עיצוב "Ephemeral" של הסביבה

לצורך כך, אומץ עיקרון של תשתית זמנית: הסביבה מוקמת מחדש בכל תחילת עבודה (terraform apply) ונהרסת בסיומה (terraform destroy), כך שהעלות נגבית רק בזמן שהסביבה אכן פעילה בפועל.

מצבעלות מוערכתמה נשאר פעיל
סביבה פעילה (running)~5$ ליוםEKS control plane, NAT Gateway, worker node אחד, Load Balancer
סביבה מושמדת (destroyed)~0.5$ לחודשHosted Zone ב-Route53 בלבד (בשכבת bootstrap)

הפרדת שכבת ה-bootstrap (הקבועה) משכבת ה-envs/dev (הזמנית) היא זו שמאפשרת מודל עבודה כזה: רק המשאבים שחייבים לשרוד לאורך זמן (כתובת ה-DNS, ה-state) אינם נהרסים.

אתגרים ופתרונות במהלך הפיתוח

אתגר: ניהול בטוח של הרשאות AWS בין מספר מפתחים בצוות
פתרון: הגדרת מדיניות IAM מותאמת אישית וממוקדת (least privilege), הניתנת לשכפול לכל חבר צוות, במקום הענקת הרשאות מנהל גורפות.
אתגר: מניעת עלויות מיותרות בסביבת פיתוח שאינה בשימוש מסחרי
פתרון: תכנון התשתית כ-ephemeral, עם הפרדת שכבה קבועה (bootstrap) משכבה זמנית (envs/dev), המאפשרת עלייה והשמדה מלאה תוך דקות.
אתגר: העברת ניהול הדומיין הציבורי מרשם חיצוני (GoDaddy) לענן AWS
פתרון: הקמת Hosted Zone ב-Route53 באמצעות Terraform, ועדכון nameservers חד-פעמי אצל הרשם החיצוני להצביע על השכבה הקבועה בענן.
אתגר: סנכרון תשתיות בענן לאורך זמן בין מספר מפתחים
פתרון: Version Control + Terraform עם state מרוחק (S3 + DynamoDB לנעילה) → שליטה מוחלטת ועקבית בשינויים.
אתגר: כיסוי דרישת ניטור לוגים מרוכז מבלי להכביד על סביבה זמנית
פתרון: הוספת Grafana Loki + Alloy כפתרון קל-משקל, המשתלב באותה Grafana הקיימת ואינו דורש PVC נפרד.
אתגר: ניהול ידני של פריסות עלול לגרום לתקלות ולחוסר עקביות
פתרון: אימוץ GitOps (ArgoCD) כמנגנון פריסה רשמי ותואם, עם Git כמקור האמת היחיד.
אתגר: פודים נתקעו במצב Pending למרות שהיה מספיק CPU/Memory פנויים בצומת

פתרון: אימות כל שינוי בעזרת helm template באופן מקומי לפני הרצת terraform apply בפועל, כדי לתפוס שגיאות סכמה מראש במקום לגלות אותן רק באמצע פריסה בענן.
